\section{Introduction}
In this note I describe the actions to be taken w.r.t superdeixis and polarity
attributes.

These conditions and actions will appear in many different rules, and
it would be nice if the treatment of these phenomena is being done uniformly.

In the ideal situation we would like to write separate functions or
procedures for the things being done here, so that a simple call of e.g. 
{\em checkandadaptsuperdeixis } or {\em adaptpolarity } applied to the relevant
attributes would suffice.

Unfortunately, this is not directly possible, for reasons we will sketch below.

The section on superdeixis is based on doc. 0053 (Lisette Appelo- Translation 
of Temporal Expressions in Rosetta3). 

The section on polarity is based on doc. 0141 ( Franciska de Jong and 
Jan Odijk- Dutch Domain T. Documentation), section 4.

\section{What to do with superdeixis?}

The following assumptions are made:
\begin{itemize}
  \item {\em  deixis} and {\em  superdeixis} are set in the deixisrules,
         that will be written by Lisette. Before that point, these attributes
         are either not present, or they have an omegavalue. 
       (Superdeixis is present in the XPPROPrecords, although it is 
superfluous there. It should be present only in the records for XPFORMULAs,
and OPEN/CLOSEDXPPROPs, but these have the same record as XPPROPs.)
  \item {\em superdeixis } can get a value by the rules specified below
        if no deixis attribute is present, or if it has the value Omegadeixis
  \item The deixis rules make every CLAUSE fourfold ambiguous: 
         \begin{enumerate}
           \item deixis is independent:\\
                 superdeixis:=Omegadeixis               
	       \begin{enumerate}
		 \item deixis:=presentdeixis
		 \item deixis:=pastdeixis
	       \end{enumerate}
           \item deixis is dependent\\
                 deixis:=Omegadeixis
	       \begin{enumerate}
		 \item superdeixis:=presentdeixis 
		 \item superdeixis:=pastdeixis
	       \end{enumerate}
          \end{enumerate}
Since the setting of deixis is translationally relevant, in fact there will
be only a two-fold ambiguity.
  \item I assume that it is specified somehow that certain of these options
        are excluded in certain environments, e.g. (without claiming reality
of these specific examples) main sentence must have
        independent deixis, relative clause must have dependent deixis etc.
If this is not specified somewhere, then it makes the superdeixis mechanism
partially superfluous, since no form is excluded anywhere. The superdeixis
mechanism is then only necessary for clauses without any overt tense
whatsoever (infinitives, participles). If the superdeixis attributes is
used for this latter purpose only, then I'd prefer the approach taken in 
Rosetta2 (where a record has either a deixis attribute, or a superdeixis attribute,
never both).
 \end{itemize}

The typical situation w.r.t superdeixis is a situation where two S-trees
are combined (speaking generatively) into one new S-tree. (For other situations
see below). The records relevant to superdeixis transmittal will be called
here {\em rec1 } and {\em rec2 }. In the output model there are corresponding
records, called {\em rec3 } and {\em rec4 }, respectively. So {\em rec3} 
corresponds to {\em rec1 } and {\em rec4 } to {\em rec2 }
This new S-tree will have a node as a topmost element. Its record
must be {\em rec4 }. 
Some examples:
\begin{enumerate}
  \item NP is substituted for a VAR in SENTENCE, resulting in\\
 SENTENCE[...NP..]: rec1=NPrec, rec2=SENTENCErec in the input models;
rec3 = NPrec, rec4=SENTENCErec in the output model.
  \item NP is substituted for a VAR in SENTENCE to form a NP modified by a
     relative clause: NP[ .. SENTENCE ]: rec1=SENTENCErec, rec2=NPrec
in the input models; rec3=SENTENCErec, rec4 = NPrec in the output model. 
\end{enumerate}


So the node associated with rec4 must dominate the node associated with rec3
in the output model, and, in fact  rec4 must be the topnode of this outputmodel.

Normally, the actions and conditions w.r.t superdeixis can be carried out upon
combining two S-trees. However, there is at least one exception to this:

The RC:Propositionsubstitution is applied before anything is known about
deixis. For that reason the conditions and actions to be carried out for
superdeixis are carried out in the PROPOKrules.

A second case where problems might occur concerns adverbials: if ADVP-s can
contain sentences with dependent deixis (e.g. {\em interessanter dan
Piet schreef} , and if such adverbials are introduced before deixis rules
(this is the case at this moment), then a similar problem arises:
(cf. {\em  Hij vertelde interessanter dan Piet schreef}).

This latter problem has not been solved yet.

In the next subsections it is described what action is to be taken w.r.t 
superdeixis.

\subsection{Generatively}

Several subcases must be distinguished, basically w.r.t.  whether rec1 or rec2

The relevant conditions and actions have been written down as condition-action
pairs.




\begin{enumerate}

\subsection { Other Situations }

The following are situations that differ slightly from the situations
described above:
\begin{enumerate}
  \item rec2 does not exist.\\
         Example: put a CN over a NOUN. Here
        rec1,rec3 = NOUNrecord; rec4 = CNrecord. 
       Normally no deixis attribute is present on the relevant records.
The actions to be taken are:
	       \begin{itemize}
		 \item \mbox{}
		       \begin{verbatim}
rec3 := rec1;
rec4.superdeixis := rec3.superdeixis
		       \end{verbatim}

	       \end{itemize}

  \item rec2 does not exist and there are {\em n} rec1's.
Example: Form a NP out of NP$_{1}$, ..., NP$_{n}$ with a CONJ
        In this case the following actions are to be taken: Do all
C-A's (to the extent that they are relevant) for each pair (rec1$_{n}$, rec4)

\end{enumerate}

\subsection{ Some changes in the Method}

There is one changes that might be added to the proposal as done now:
\begin{enumerate}
  \item adopting the (what might be called) 'Mountain Climber Method': This
is the method adopted in R2, where rec3.superdeixis is always set to Omega
at the end of a rule. The major advantage of this method is that the
surface parser need not take care of superdeixis.
\end{enumerate}

This method should be worked out somewhat more.

\section{How to use all this}

Ideally, we would like to write a separate procedure to take care of
superdeixis, and polarity. A problem with this approach, however, is formed
by the fact that the CA-pairs of superdeixis and polarity can create 
ambiguities (in particular, analytically). Given this fact it is not easy to 
rebuild these CA-pairs as procedures.

Th\'{i}s problem could be solved by making an additional procedure
for each case where an ambiguity can arise, and calling these procedures
in separate condition action pairs, e.g. splitting up (1a) of the superdeixis
case into two procedures, gensuperdeixis1a1 and gensuperdeixis1a2, that
are applied to the parameters (super1,super2:deixistype; VAR super3, super4:
deixistype), where gensuperdeixis1a1 applies all CApairs (in the form of IF
THEN statemnets) except C1A1, and gensuperdeixis1a2 applis C1A1. It must be
assumed here that the copying of the records has been done independently.
The procedures can be called in the following way:
\begin{verbatim}


         C1: ...(* other conditions *)
             C11: true
             A11: rec3 := rec1;
                  rec4 := rec2;
                  gensuperdeixis1a1(rec1.superdeixis, rec2.superdeixis,
                                    rec3.superdeixis, rec4.superdeixis)
             C12: true
             A12: rec3 := rec1;
                  rec4 := rec2;
                  gensuperdeixis1a2(rec1.superdeixis, rec2.superdeixis,
                                    rec3.superdeixis, rec4.superdeixis)
         A1: (* other actions *)
\end{verbatim}

By nesting polarity CA pairs inside these, also these can be accounted for.

The problem with this approach, however, is the following:
If the conditions inside the special procedures are not met in any case , 
then a new structure is created {\em without any adaptations w.r.t. superdeixis
and/or polarity}, which is the wrong result: In that case the rule should not
be applicable, i.e. not yield any structure whatsoever.

Another option  would be to make separate transformations following 
substitution rules that account for superdeixis, and for polarity.

The disadvantages of that approach, however are:
\begin{enumerate} 
  \item It is a rather complex mechanism, used only to check whether the values
        of certain attributes have the appropriate value, or to adapt these 
        slightly. 
  \item It is not guaranteed that the same conditions and actions w.r.t 
superdeixis and polarity are applied again and again
  \item If these transformations should have to apply independently of
        substitution rules in certain cases (see section 2), and if they have
        to apply iteratively, then a measure condition prioblem will arise.
        (Procedures can be included in other rules for which there is no 
        problem w.r.t. the measure conditions)
\end{enumerate}

At the moment I see no satisfactory way to resolve these disadvantages and
problems adequately. All suggestions are welcome.
