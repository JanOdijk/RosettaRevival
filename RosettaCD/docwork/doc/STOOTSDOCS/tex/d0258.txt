\section{Introduction}
At the moment variables in Rosetta3 are treated as described in document R0159 
(J. Landsbergen - Variables and pre-actions). A consequence of that approach
is that various VAR-categories must be introduced in domain T. In order to
have information about the S-tree that is to be ( or, has been ) substituted for the
variable, the variable is given a set of attribute value pairs in which this
information can be stored. Both in analysis and generation this approach
can be justified theoretically. Apparently, however, there is a need for
more detailed information about the S-tree that is substituted, for instance
information about its internal structure. Of course, some of this information 
could also be stored in attributes, but this seems very impractical. 

\section{The new proposal}
The idea behind the proposal is very simple. It is based on an extension of 
the M-rule syntax. First, the set of variable categories is reduced to one 
single category; the {\bf VAR} category.
Furthermore, the only attribute in a variable record will be the variable 
index. 
Next, the M-rule syntax of S-tree model specifications is extended as follows. 
It wil be allowed to enrich a variable which occurs in an M-rule
model with a condition on the S-tree
that has been ( during analysis ) or is going to be ( during generation )
substituted for the variable:

\begin{verbatim}
    Example

    < m: CLAUSE{CLAUSErec1}[mu1,rel1/VAR(index1)/NP{NPrec1}[mu2]]
    >

\end{verbatim}

In this example the condition on the variable says that the substituent
associated with the variable has to be of category {\bf NP}. The conditions
on the substituent can also be more complicated.
One can imagine that the rule-writer wants to put
conditions on the value of the variable {\bf NPrec1} or on the structure of 
{\bf mu2} by means of a match-condition. In general, all constructions
which are allowed in normal models are also allowed in conditions on 
variables.

The condition on the variable is ignored in the left models during
analysis and in the right models during generation. This implies that the
substituent associated with the variable cannot be changed.

\section{Theoretical justification}
I think that a theoretical justification is possible if we use an extension
of the strategy used in the theoretical interpretation of the current variable 
treatment. In the current
treatment variables can carry, besides their index, all kinds of syntactic
information ( syntactic category, person, number etc. ). The extension is to 
allow variables to carry all the other information stored in an S-tree as well.
The ony real difference with the current treatment is that the notion 
\underline{infinity} sneaks into the definition. 
Here, I will assume that the set of possible syntactic 
variables is defined as the subset T$_{v}$ of the set {\bf T} (cf. R0011) such
that T$_v$ contains all the S-trees with topcategory {\bf VAR}. Using this 
extension the variable treatment is as follows:
\begin{itemize}
   \item In A-TRANSFER a syntactical variable is translated into a logical
variable; during this step all information except the index is lost.
\end{itemize}
\begin{verbatim}
    Example

             VAR(i)
              |
             NP          ------>   Xi
             /\
            /  \
           /____\     

\end{verbatim}
\begin{itemize}
   \item In G-TRANSFER a logical variable is translated in an (infinite) set of
         strees. The only restriction on the S-trees in the set is that their 
         topnode is of category {\bf VAR} with the index of the logical 
         variable.
\end{itemize}

\begin{verbatim}
    Example

               VAR(i)    VAR(i)     VAR(i)
                |          |          |
    Xi  ---> {  NP     ,   UTT    ,  BVERB , ........... }
                /\                   /\
               /  \                 /__\
              /____\     

\end{verbatim}
\begin{itemize}
   \item The M-rules operate on the fully specified variable. Therefore, 
theoretically, the condition on variables in M-rule models which was introduced
in the previous section is a condition on the son S-tree of the VAR-node.
   \item The generative variable substitution rules have to check whether 
the S-tree which is substituted for the variable is the same as the son S-tree 
of the actual variable. The substitution is only allowed if the S-trees are 
equal.
   \item In the analytical variable substitution rule a variable node is 
created. Furthermore a copy of the complete substituent is connected as the
son S-tree of the variable node.
\end{itemize}

\section{Implementation}
In the implementation of this proposal a variable will be simpler than the 
variable in the implementation of doc R0159. A
syntactic variable is implemented as one single S-tree node. The only variable 
category is the category {\bf VAR}, and furthermore, each variable node has 
just one attribute, namely the index of the variable.

The preaction mechanism in the M-generator is simplified in the sense that
it just has to take care of the correct evaluation order of the sons of
a variable substitution rule node. Furthermore, it must make the result of this
evaluation "globally" available as a function of the index of the variable 
substitution rule ( because the conditions on the variables will also be 
allowed in analysis, M-Parser must provide a similar function ).

If the M-rule compiler meets a condition on a variable, it will generate
PASCAL code in which the following actions can be distinguished:\\
\begin{enumerate}
\item GET the substituent stree associated with the variable using its index.
\item APPLY the variable condition to this S-tree.
\end{enumerate}

\section{Realization}
There is one major problem which prevents a straight-forward realization 
of the variable treatment proposed in this document. In the 
current M-grammar complications arise because the value of some 
variable attributes is changed during the parse process for reasons of
efficiency of notation. At first sight 
this abuse makes
it impossible to introduce the new variable treatment whithout considerable
changes in the M-rules. There is a solution, however, the realization
of which requires minimal effort for lingo's as well as for sopho's. Using
this solution the proposal can be realized as follows:

\begin{itemize}
   \item The introduction of a new subgrammar for variables.
         This subgrammar contains one rule or subrule for each variable
         category, analogous to the following example:
\end{itemize}
\begin{verbatim}

     Example:

           <m1: VAR(index1)/NP{NPrec1}[mu1]
           >
           <m : NPVAR{NPVARrec1}[head/VAR(index1)/NP{NPrec1}[mu1]]
           >

           COMP
              < C1: TRUE
                A1: NPVARrec1 :=  COPY_NPtoNPVAR(NPrec1);
                    NPVARrec1.index := index1;
              >

           DECOMP
              < C1: NPVARrec1 = COPY_NPtoNPVAR(NPrec1) AND
                    NPVARrec1.index = index1
                A1: @
              >

\end{verbatim}        
\begin{itemize}
     \item The models of the rules in which these variable categories are used
           will have to be changed slightly because the variable nodes are
           no longer terminals. Therefore, instead of \\
           NPVAR\{NPVARrec1\}\\
           we have to write:\\
           NPVAR\{NPVARrec1\}[mu1].
     \item Finally, the variable substitution rules have to be changed
           according to the following example:\\
\end{itemize}
\begin{verbatim}

     Example:

           <m1: CLAUSE{CLAUSErec1}[mu2,
                                   NPVAR{NPVARrec1}[VAR(index1)],
                                   mu3
                                  ]
           >
           <m2: NP{NPrec1}[mu1]
           >
           <m : CLAUSE{CLAUSErec1}[mu2,
                                   NP{NPrec1}[mu1],
                                   mu3
                                  ]
           >
           PARAMETERS
           <
              LEVEL
           >

           COMP
              < C1: NPVARrec1 = COPY_NPtoNPVAR(NPrec1) AND
                    index1 = LEVEL AND
                    NPVARrec1.index = LEVEL
                A1: @
              >

           DECOMP
              < C1: TRUE
                A1: NPVARrec1 :=  COPY_NPtoNPVAR(NPrec1);
                    index1 := LEVEL;
                    NPVARrec1.index := LEVEL;
              >

\end{verbatim}

It will be clear from these examples that the changes needed in the existing
M-rules in order to realize the new proposal are marginal. 
\section{Conclusions}
The new proposal described in this document offers the possiblity to  
use inside an M-rule information of the S-tree that is to be substituted 
for a variable. 

Furthermore, there is only one variable category {\bf VAR} which is not 
user-defined but imposed by the formalism. 

Finally, the pre-action technique is simplified, in fact it is simplified to
such a degree that the word "action" in pre-action is somewhat exaggerated. The 
main difference between the old and new pre-action is the fact that no copy 
action exists in the new proposal. The copy component of the pre-action copies
the topnode record
of the substituent S-tree to the record of the variable. Because these records
are not of the same type a specification is needed of the relation between 
variable categories and the categories of the S-trees that can be substituted
for them. In this way the pre-action technique which is currently applied is not
restricted to the implementation alone, it has also its consequences for the 
linguistic specification of the system.
