\section{Introduction}
In this document the Rosetta3 bonussystem is described. The Rosetta3 bonussystem
must be looked at as a minimal system to order the translations of a
sentence in case of ambiguities.  This ordering is not perfect, as will be made clear in the following sections. In fact the bonussy
stem is only a
method to find the {\em best} translation. A consequence of the method is
that the complete set of translations is ordered.
The strategy
followed in the Rosetta3 bonussystem is that the  translation
with the highest bonus value at each system level is, eventually, the best 
translation. Of
course, this strategy may fail in some cases. It assumes, for instance, 
that a choice made in generation
on the basis of bonusses assigned during A-Transfer cannot be overruled
by bonus values assigned in the M-Generator. This can be overcome by developing
a {\em bonus evaluation system} which is able to compare different kinds of
bonusses. Research into such a system, however, was considered to be part
of the Rosetta4 phase.
\section{Bonusspecification}
At the moment bonusses can be specified at 3 places in the system:
\begin{enumerate}
\item In the Lexicon. 
\item In M-Rules.
\item In Transfer rules.
\end{enumerate}
{\bf Ad 1} Although bonus values can be specified in the lexicon for
analysis and generation purposes( cf. document 305),  only the bonusses that 
order alternatives in the mkey $\rightarrow$ skey translation  are
used by the Rosetta3 system. During analysis the system uses the order in 
which the meaning ambiguities of a word occur in the lexicon. \\
\ \ \ \\
{\bf Ad 2} In the M-rules it is possible to assign a bonus value to each 
           S-tree ( in generation) or tuple of S-trees ( in analysis) which
           is the result of an M-rule application . \\
\ \ \ \\
{\bf Ad 3} To each transfer rule specification a bonus can be added. This
           bonus is used to order the set of M-Rules which is the result
           of translating an IL-rule.\\
\section{Bonuscalculation}
\begin{description}
\item[M-Parser] The bonus specified in an M-Rule is assigned to the 
                corresponding node in the syntactic derivation tree. The 
                bonus value of the syntactic derivation tree is the sum of the
                bonus values associated with its nodes.
\item[A-Transfer] No bonus value is calculated. In general, each M-rule has
                  just one corrsponding IL-rule. Therefore, there is no urgent
                  need to discriminate between alternatives at this level.
                  Furthermore, interaction with the user is used to disambiguate
                 at the level of basic expressions.
\item[G-Transfer] The bonus specified in the Transfer rule is assigned to
                the corresponding node in the syntactic derivation tree. The 
                bonus value of the syntactic derivation tree is the sum of the
                bonus values associated with its nodes.
\item[M-Generator]  Again, the bonus value of the resulting S-tree is the
                sum of all bonus-values specified in the M-rules which have
                been applied to create the S-tree.                
\end{description}
If we represent these 3 kind of bonusses by means of a vector $( i,j,k)$, where
$i$ is the M-Parser bonus, $j$ is the G-Transfer bonus and $k$ is the
M-Generator bonus then the bonus ordering is defined as follows: \\
\begin{tabbing}
aaaaaaaaaaaaaaaaaaa\=aaaaa\kill
$(i,j,k) > (l, m, n)$ if $(i > l) \vee $\\
\>$((i = l) \wedge (j > m)) \vee $\\
\>$((i = l) \wedge (j = m) \wedge (k > n))$
\end{tabbing}  
\section{Complications}
The bonussystem described in this document has certainly its drawbacks.
For instance, by definition, the bonus of a  tree is the sum of the bonusses 
of its
nodes. Consequently, the bonus value of a tree is influenced by the number of 
its nodes, whereas it is not at all obvious that big trees are better (or
worse) than small trees.

Another problem are the bonusses which are assigned automatically
to the alternative 
interpretations of basic expressions. Suppose, for example, that the
basic expression `pijp' has three meanings, `pijp-1',  `pijp-2' and
`pijp-3'. If these meanings have bonusvalues 3, 2 and 1, respectively, it
means that `pijp-1' is more plausible than `pijp-2', etc.. Now, let there be
another basic expression `hoofd' with four meanings, with bonus values 4, 3,
2 and 1. According to the ordering described in the previous section
the combination `pijp-2' and `hoofd-2' has bonus 3+2 = 5 and is therefore
better than the combination `pijp-1' and `hoofd-4' which has bonus 4. It
will be obvious that this is nonsense. We cannot use the bonusses that
are assigned in this way to make such comparisons. The only thing which is
correct is that the combination `pijp-1', `hoofd-1' is better than all
other combinations. 

In previous bonus proposals( cf. r0125) the notions {\em weak} bonus,
 {\em strong}
bonus and {\em correctness} bonus were introduced. They have been 
invented in order to be able to make subtle choices between idiomatic
and non-idiomatic readings etc. The reason that these bonusses have been
omitted in this proposal is that they can easily be implemented with the
current simple bonussystem, by giving each kind of bonus a certain value
range. For instance, weak bonusses can vary between 0 and 10,  strong
bonusses have a value between 1000 and 10.000 and correctnes bonusses
have a value between 1.000.000 and 10.000.000. This implementation
of the subtle bonussystem preserves the ordering which was proposed
in document 125.
