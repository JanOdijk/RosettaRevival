\section{Introduction}
In this document a framework will be proposed for the Rosetta3 semantic
 component.
The proposed framework  contains ideas
that emerged from discussions with several 
project members (Jan O., Lisette A., Ren\'{e} L.) during the past year. 
It should be stressed that the document contains only the description
of a framework, no type-system, no sort system or semantic paradigm like
situation semantics or possible world semantics is elaborated. The framework
will be as general as possible, that is, it  allows the "implementation" 
of most of the current ideas in this field which are based on compositionality.

\section{Informal introduction of the framework}
Until now the meaning rules occurring in semantic derivation trees
have no corresponding definition. This will be changed in the new framework; 
meaning rules will be n-ary functions that operate on a special kind
of structures, say M-structures. The result of the function application will
be a set of M-structures. The set of
{\em possible} M-structures will be called $Dom(M)$ and must be specified
explicitly. In this document I will not prescribe what an M-structure
should look like. At the moment I am not sure whether the definition of
M-structures should be constrained by the formalism, as it is the case with
S-trees, or not. If there are no formal restrictions on M-structures, 
they will be completely `linguist definable', that is, the linguist will have
the freedom to define an M-structure as a tree, as one feature bundle or even
as a directed graph. The general definition of the semantic framework, as it is
presented in this paper, however, does not depend on the exact definition of the
M-structures. 

The semantic component will be very similar in structure to the M-Generator
component. Both components evaluate a derivation tree in a bottom up
way, applying the rules which are recorded in the derivation tree. A derivation
tree is correct or well-formed if the evaluation yields a non-empty
set of S-trees ( in the case of M-Generator) or M-structures ( in the case
of the semantic component). At first sight there is at least one 
difference, however. Rule application in the M-Generator is guided
by a control expression. This control expression also indicates at
which places in the process transformations may be applied.  The introduction 
of such a control expression in our semantic 
component seems not a bad idea. The same arguments can be used now as were
used in the past for the introduction of a controlled M-grammar: the 
expression makes the structure which is present in our `semantic grammar'
explicit. One can argue that, although it might be useful to specify 
control expressions over semantic rules, it is not necessary to use the
control expression during evaluation of the semantic derivation tree. However,
we can also think of another reason for control expressions in the semantic
component. The equivalent of syntactic transformations in the semantic
component would be `semantic transformations'. Semantic transformations can be
defined as meaningful rules which correspond with a syntactical identity
transformation in the 
M-grammar. Because an identity rule only passes the incoming S-tree 
without any condition, it can just as well be omitted. Then, in the semantic
component the control expression specifies at which points in the
evaluation process a semantic transformation must be inserted. 
At this moment control expressions and semantic transformations in the
semantic component may seem somewhat overdone, but introduction of these
concepts is more or less for free because of our experience with similar
constructs in M-Parser and M-Generator. 

At the moment the dictionary ILLex contains  a set of
meaning keys. Furthermore, ILLex is not used by the Rosetta system, it is
only accessed by the dictionary
compiler. This will be changed if we add the
semantic component to the Rosetta system. ILLex will then contain the set
of {\em basic M-structures}. A basic M-structure will probably be a set
of attribute value pairs, with a {\em key} attribute, an attribute in
which the {\em sort} of the M-structure is specified, e.g. {\em human, 
animal }, an attribute for the type of the basic meaning, e.g.
$<$e$>$, $<$e,t$>$ etc., and other semantic attributes, for instance for the
treatment of {\em time}. 

In fact there are two obvious processing strategies for the semantic component.
One approach is according to  the Rosetta tradition: at first make things
ambiguous and filter later on ( to be called the AF approach). The other
approach is 
to keep things vague or unspecified at first and to calculate the correct 
values later  ( we call this approach the VC approach). 

In the AF approach A-Transfer generates all possible semantic derivation
trees. If there are semantic rules with parameters and if it is not known
in A-Transfer what the exact value of these parameters must be ( for instance,
because it depends on other semantic information in the tree )   then the
corresponding syntactic rule is translated in a set of semantic rules with 
all possible parameter values.
The task of the semantic component is then to  filter out the correct semantic
derivation tree by applying bottom up the meaning rules which are recorded in 
the semantic derivation tree. The parameter mechanism which deals with
the actual transfer of the rule parameter values to the
rule can be the same  mechanism as is used in the compositional
version of the M-rules (in M-Generator).
The reversability condition
with respect to the Transfer functions is obeyed by the AF approach: 
G-Transfer is the reverse of A-Transfer. 

The second approach, the VC approach, implies that A-Transfer
does {\bf not} generate all possible semantic derivation trees. In stead,
some syntactic rules that correspond with parameterized semantic rules,
will have a `vague' translation. In cases where with the AF approach all
 possible
parameter values will be generated when translating a syntactic rule
in the corresponding parameterized semantic rule, the VC approach generates 
just one translation, namely, the corresponding semantic rule with an 
unspecified
parameter value. The correct value of the parameter will be calculated
during evaluation of the semantic derivation tree in the semantic component.
Here too, we can use an already existing mechanism. Assignment of a value to a
parameter in the semantic component can be done in the same way as the
value assignment of parameters in the decompositional version of the M-rules
(in M-Parser). 
Notice that in the VC approach G-Transfer is not the reverse of A-Transfer.  
\section{Formal definitions}
In the Rosetta system the semantic component ultimately defines the set of 
well-formed semantic derivation trees. The semantic component doesn't give an 
independent definition of the {\em interlingua}, however. The set of well-formed
interlingua trees is defined by the syntactic, the transfer and the semantic
component together.\\
Let $D_M$ be the set of well formed syntactic derivation trees, defined
by the syntactic component of the M-Grammar and let $I_{Tr}$ be the
set of semantic derivation trees defined as\\
\ \\
$I_{Tr} = _{def} \{ i \mid \exists d \in D_M: i \in $A-Transfer$(d) \} $ \\
\ \\
Then, the set of well-formed semantic derivation trees $M_M$ is defined as\\
\ \\
$ M_M =_{def} \{ i \in I_{Tr} \mid $M-Sem$(i) \neq \emptyset\} $\\
\ \\
The function M-Sem is defined by the semantic component, its definition 
will be given later. The semantic component also consists of :

\begin{enumerate}
\item The definition of the set $M$ : the domain of possible M-structures
\item The set $M_B$ of basic M-structures, $M_B \subseteq M$. The set $M_B$
       is also called ILLex.
\item  A set of semantic subgrammars SEM-SG, of which each subgrammar SG$_i$
       defines
   \begin{itemize}
     \item a set SEMRULES$_i$ of meaning rules $I_k$. A meaning rule $I_k$ 
       defines an n-ary function $F_{I_{k}}$ of type $M^n \rightarrow 2^M$.
     \item a set SEMTRFS$_i$ of semantic transformations  $J_l$. A 
       transformation $J_l$ defines an unary function $F_{J_{l}}$ of type
       $M \rightarrow 2^M$.
     \item a regular expression over the rules defined by the subgrammar.
    \end{itemize}
\end{enumerate}

In this discussion paper we will only give the definition of a simplified
version of M-Sem (according to the AF approach) : the version without 
subgrammars and 
semantic transformations. This simplified version suffices to illustrate the
functioning of the semantic component.
\begin{tabbing}
aaaaaaaa\=aaaaaaa\=aaaaaa\=aaaaaa\=aaaaaa\kill
M-Sem$(i) =_{def}$\\
\>$\{i \mid i \in ILLex \} \; \cup $ \\
\>$\{m \mid i = I<i_1,\ldots,i_n> \wedge $ \\
\>\>$\forall j: 1\leq j\leq n: m_j \in $M-Sem$(i_j) \wedge $ \\
\>\>$m \in F_I(m_1,\ldots m_n) \}$ 
\end{tabbing}
\section{Implementation Considerations}
We mentioned before that most of the concepts needed for the formalism of
the semantic component are not new within the Rosetta system. 
\begin{itemize}
\item The function M-Sem is very similar to the M-Generator function 
\item The rule parameter passing is similar to the parameter passing in M-Parser
       or M-Generator. 
\item The fact that the semantic component consists of a domain definition
      and of rule definitions operating on structures defined in the domain
      is similar to the situation in the syntactic component.
\end{itemize}
For the software this has the advantage that previously developed modules
can be reused with minimal modifications. We can expect that the following
effort is needed:
\begin{enumerate}
\item Implementation of M-Sem. Can be done on the basis of the current 
      M-Generator implementation or on the Rosetta2 implementation of
      M-Generator if it is decided that we can do without control
      expressions and transformations in the semantic component.
\item Implementation of a semantic domain compiler. The current syntactic
      domain compiler can be used as starting point if it is decided
      that M-structures will be similar in structure to S-trees, that is,
      if M-structures will be tree-like structures with labeled nodes.
      The semantic domain
      compiler will then probably be a stripped version of the syntactic
      domain compiler.
\item Implementation of a semantic rule compiler. If we choose M-structures
       to be similar to S-trees then we can probably define the semantic rule
      language to be the same as the M-rule language. In that case we
      can very easily generate a new compiler by means of the
      compiler generator and the M-rule language specifications. Next,
       new language constructs, e.g. the sort inclusion test, can be added. 
\item Extension of the current ILLex compiler in order to be able to cope
      with the specification of attribute-value pairs. 
\item Implementation of the abstract datatype {\em M-structure}.
\item If we decide to use a taxonomy, a network of concepts, we will
      need a language in which the taxonomy can be specified and a
      compiler which translates this description into PASCAL code.    
     
\end{enumerate}
\section{Conclusions}
This paper only deals with the changes of the formalism and with the
implications of the introduction of a semantic component for the Rosetta
software. We realize that also major changes are necessary in the lingware of
Rosetta3 if it is decided to move the treatment of some concepts from
syntax to semantics. These lingware modifications involve alterations in
M-rules and lexicons. Changes in the lexicon imply that information which
is now stored in BLEX can be moved to ILLex. This will have a favourable effect
on the efficiency of the system because  entries which only differed with
respect to semantic information can be joined to one single entry.
