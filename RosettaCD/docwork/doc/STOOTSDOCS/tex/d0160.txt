\section{Introduction}
In this note I want to discuss some of the implementation decisions
that Carel Fellinger and I made with respect to the separation of language
specific and language independent software. In Fellinger(R018, page 5-7) and
Rous( Programmatuur voor het vertaalsysteem Rosetta, page 24) is described
how the question has been dealt with in Rosetta2 and Rosetta1 respectively.
\section{The Rosetta3 proposal}
In the definition of the Rosetta formalism one can clearly distinguish 
two different kinds of functions; functions wich are the same for all
languages, e.g., M-PARSER, M-GENERATOR, ANALYSIS, etc., and functions
which are different for all languages, e.g., the lextree rules, the M-rules,
etc.. We are convinced that this distinction also should be made in the
software of the Rosetta system. 

The solution we propose for Rosetta3 is,
in our opinion, better than the Rosetta2 solution (and of course much better 
than the Rosetta1 solution), but it is not the final solution we have in mind. A
prerequisite for the treatment of language specific software (to be called
LS software) and language independent software (to be called LI software) that 
we propose is a programming language with a Modula-2 like module mechanism.
The language used in the Rosetta project, as described in Fellinger-Rous(R64),
supplies such a mechanism. The final solution we have in mind can only be 
implemented in a programming language with a ADA-like generic mechanism.
Before I start discussing our proposal, I'll first try to sketch in a few 
words the module mechanism that we use. 

A module consists of a definition
module and an implementation module. The definition module contains only
those declarations that are exported by the module. The things specified in 
the definition module are in principle visible for other modules, i.e. module
{\bf A} can use the definitions of module {\bf B} by explicitly stating in the
implementation and/or definition module that it {\em inherit}s {\bf B}.
One of the major advantages of this mechanism is the amount of recompilation
time reduction. If the implementation module changes only the implementation
module has to be recompiled. However, if the definition module changes
each module which inherits that definition module has to be recompiled,
including the corresponding implementation module.

The next three steps seem to be a useful and straightforward application of the 
module mechanism to the M-PARSER function.

\begin{enumerate}
\item Create a module {\em M\_PARSER}, in which the function M-PARSER is 
implemented.
\item Create a module {\em M\_RULES}, in which the M-rules are implemented.
\item The module {\em M\_PARSER} should inherit the module {\em M\_RULES}.
\end{enumerate}

Unfortunately these steps do not lead to a solution of our problem.
This is caused by the fact that there not just {\em one} {\em M\_RULES} 
module, but one 
for each language. Therefore it is necessary to specify inside the 
{\em M\_PARSER} 
module which module {\em M\_RULES} ( Enlish, Dutch or Spanish ) must be 
inherited. 
Therefore a copy of the module {\em M\_PARSER} must be created for 
each language, each one inheriting the {\em M\_RULES} module of the language
under consideration. Consequently, we do not have a language independent module 
{\em M-PARSER}.

\subsection{A solution for functions and procedures}

The situation sketched above occurs very frequently in the Rosetta system, 
in fact each time when a (potential) LI function calls a LS function. For
Rosetta3 we therefore propose to define a sofware layer that acts as an
interface between the LI and the LS software. This software layer should
consist of modules of which the definition module is LI and the implementation
module is LS. We say that this layer contains language dependent software
( to be called LD software ). 

For the {\em M\_PARSER} case this means
that a module, say, {\em M\_RULESWITCH} is defined, which has a LI
definition module part and a LS implementation module part. The task of the
implementation module is to call an individual M-rule. The inherit relation
is now as follows: 
\begin{itemize}
\item The implementation module {\em M\_PARSER} inherits the 
{\em M\_RULESWITCH} definition module.
\item The implementation module {\em M\_RULESWITCH} inherits the definition
module {\em M-RULES}.
\end{itemize}

This implies that it is not necessary to have a copy of module {\em M\_PARSER}
for each language and, consequently, that module {\em M\_PARSER} is 
legitimate LI software. Another consequence of the above construction is
the fact that {\em M\_PARSER} needs not to be recompiled if a change inside
an arbitrary M-rule has taken place.

\subsection{A solution for datastructures}

If we just consider parameterless functions and procedures the solution 
presented in the
previous section is sufficient. However, functions and procedures do have
parameters that enable us to pass information. As we will see, this information
can also be LS or LI. 

The M-rules in the example of the
previous section operate on S-trees and they return a set of S-trees to the 
calling function. Via the module {\em M\_RULESWITCH} this information 
eventually ends up in the {\em M\_PARSER}. But because these S-trees are 
highly LS, the module {\em M\_PARSER} also becomes LS because it must inherit
the module in which S-trees are defined, and ... we are back were we started.
To solve this problem we can use the LD software layer described in the
previous section. Because the LD layer acts as an interface between LI and
LS software, all information which is exchanged between LI and LS parts 
is excanged via the LD layer. 

Our problem is solved if we are able to translate LS information into LI 
information before it is passed from LS sofware parts to LI software parts and
vice versa. The LD layer certainly seems the best place to perform such a 
translation.

In our example this would mean that the module {\em M\_RULESWITCH} translates
an LS S-tree datastructure into an LI S-tree datastructure. This step, however, 
makes only sense because 
LI software never refers to the actual value of information 
which is returned from LS software parts.
One can argue that it must be possible for LI software to 
perform, for instance, a check on 
equality between two parts of LS information. In that case, however, the LI 
software is not interested in the {\em absolute} value of the LS information but
rather in the {\em relative} value with respect to other LS information.

The only correct solution of this problem in {\bf Rosetta Pascal} would be to 
make a bijective translation 
function that can translate a piece of LS information into a set of integer, 
real and/or boolean values. This translation function has to be applied 
by the LD software each time information is exchanged between LI and LS 
software. This solution would be, although conceptually correct, in practice 
rather inefficient and timeconsuming. 

The solution that we propose is much more efficient but unfortunately not
correct Rosetta Pascal. On the condition that two variables containing 
corresponding pieces of LS and LI information have the same alignment in 
computer memory 
they can in principle be assigned to each other if it would be possible to 
switch off the typechecking mechanism during the assignment. Because VAX/VMS 
PASCAL offers
this possibility ( type cast operator ), we propose to implement the 
translation function by means
of this VMS specific feature. Of course we realize that some people 
shudder at the mere thought of a type cast operator but there are a number of
extenuating circumstances. Firstly, there is a correct alternative method that
can be used in the very unlikely case that Rosetta must be transported to 
another operating system and
secondly, the use of the type cast operator is restricted to the LD sofware
layer which for the most part is generated automatically by our compilers. 

In the next section I will work out in detail the consequences of this
solution for the {\em M\_PARSER} example and in particular for the main
datastructure, the S-tree.

\subsection{Consequences for the M\_PARSER Example}
In Rosetta3 we will have for each language the modules LSDOMAINT and 
LSSTREE. LSDOMAINT contains the declaration of the syntactic categories
( LSDOMAINT\_syntcat ), the syntactic relations ( LSDOMAINT\_synrel ) and
for each category its corresponding record definition. The module is generated
by the {\em domainT-compiler}, which always generates a fixed number of
syntactic categories and relations. If the number of categories specified
by the linguist is less than this fixed number, the compiler completes the
number by adding dummy categories, cq relations.

The LSSTREE module inherits the LSDOMAINT module and 
contains the structural definition of the LS S-tree:\\

\begin{verbatim}
TYPE
   LSSTREE_pStree   = ^LSSTREE_Stree;
   LSSTREE_pRelNode = ^LSSTREE_RelNode;

   LSSTREE_Stree    = RECORD
                         cat : LSDOMAINT_syntcat;
                         sons: LSSTREE_pRelNode;
                         LI  : LISTREE_pLIrecord;
                         LS  : LSDOMAINT_pLSrecord
                      END;
   
   LSSTREE_RelNode  = RECORD
                         brother : LSSTREE_pRelNode;
                         node    : LSSTREE_pStree
                         relation: LSDOMAINT_synrel;
                      END;

\end{verbatim}
The {\em LI}-field of the S-tree record definition contains LI information, 
like the number of sons etc. The {\em LS}-field is a pointer to a LS record 
containing the LS information corresponding to the syntactic category 
specified in the field {\em cat}.

In addition to the LS modules LSSTREE and LSDOMAINT that have been defined for
each language, we have the LI modules LISTREE and LIDOMAINT. 
LIDOMAINT contains the declaration of the LI syntactic categories
( LIDOMAINT\_syntcat ) and the LI syntactic relations ( LSDOMAINT\_synrel ) 
The number of LI categories and LI relations will be exactly the same as
the fixed number LS categories and relations generated by the domainT 
compiler, to be sure that they occupy the same amount of memory.

The LISTREE module inherits the LIDOMAINT module and 
contains the structural definition of the LI S-tree, as follows:\\

\begin{verbatim}
TYPE
   LISTREE_pStree   = ^LISTREE_Stree;
   LISTREE_pRelNode = ^LISTREE_RelNode;
   LISTREE_pLIrecord= ^LISTREE_LIrecord;

   LISTREE_Stree    = RECORD
                         cat : LIDOMAINT_syntcat;
                         sons: LISTREE_pRelNode;
                         LI  : LISTREE_pLIrecord;
                         LS  : ^INTEGER
                      END;
   
   LISTREE_RelNode  = RECORD
                         brother : LISTREE_pRelNode;
                         node    : LISTREE_pStree
                         relation: LIDOMAINT_synrel;
                      END;

   LISTREE_LIrecord = RECORD
                         N       : INTEGER;
                         Dtree   : LIDTREE_pDtree
                      END;

\end{verbatim}
The LSSTREE\_Stree and LISTREE\_Stree types are very like as we can see from 
these definitions. The precautions we took guarantee that the 
mapping to memory of the LSSTREE\_Stree and LISTREE\_Stree type is the same.
Below I've elaborated the MRULESWITCH example.

\begin{verbatim}
DEFINITION MODULE LD_MRULESWITCH;

   INHERITS LISTREE;

   EXPORT
   PROCEDURE MRULESWITCH(i: INTEGER; t: LISTREE_pStreeTuple): LISTREE_pStreeSet;

END {LD_MRULESWITCH}.



IMPLEMENTATION MODULE LD_MRULESWITCH;

   INHERIT LISTREE, LSSTREE;

   PROCEDURE MRULESWITCH(i: INTEGER; t: LISTREE_pStreeTuple): LISTREE_pStreeSet;
   VAR
      LSs: LSSTREE_pStreeSet;
      LSt: LSSTREE-PStreeTuple;
   BEGIN
      (*-------------------------------------------------------------*
       * During the next assignment, we cast the type of parameter t to
       * the language specific type LSSTREE_pStreeTuple ( by means of the
       * type cast operator ::) , which enables us
       * to assign the value of t to a variable of type LSSTREE_pStreeTuple.
       *------------------------------------------------------------*)
      LSt := t::LSSTREE_pStreeTuple;

      CASE i OF
      .
      .
      7 : LSs := NOUNrule(LSt);
      .
      END;
      (*------------------------------------------------------------*
       * The interface of the function MRULESWITCH is language independent.
       * Therefore we cast the type of variable LSs to the language independent
       * type LISTREE_pStreeSet. 
       *-----------------------------------------------------------*)
      MRULESWITCH := LSs::LISTREE_pStreeSet
   END {MRULESWITCH};

END {LD_MRULESWITCH}.
\end{verbatim}

The example shows that at the entrance of the function MRULESWITCH the type
of the LI parameter is casted to a LS type so that it can be passed to the
actual M-Rules. At the end of the function the type of the LS variable
is casted to the LI result type of the function.
