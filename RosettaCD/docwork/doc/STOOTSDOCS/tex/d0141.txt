\section{Introduction}

This document is the documention for DUTCH DOMAIN T. In this document DOMAIN T
for Dutch itself is not given. For this, see document no. R0146. 

This document 
gives indications concerning the intended
use of categories, records, relations, attributes and their values, etc..
This document incorporates the documentation of a modified version of DOMAIN T
for morphology as described by Harm Smit in document 0134. The syntax of that
description has been adapted to the requirements of doc. 0128, and some changes
in the content have been made, after consultation with Harm Smit and the other
morphologists. This document overrules the DOMAIN T part of document 0134. 

It is possible that a study of certain 'closed' classes will require
some further extension of this domain. In particular, the class of pronouns
that appear in VanDale do not all fit into the categories distinguished here.
Jan Odijk is preparing a separate note on this.

\section{General remarks}

Typenames, Categories and Records have been listed in alphabetical order,
where the prefixes {\em B- }, {\em SUB- }, {\em OPEN- }, {\em CLOSED- } and
{\em R- } (as in {\em RADV }) have been ignored. 
All VARcategories are alphabetized under {\em V }. The category {\em CN }is
listed under {\em N }. {\em EN } is listed under {\em E }

The following naming conventions have been adopted:
\begin{itemize}
  \item English names for attributes and values have been used.
  \item SET attribute names always end in 's'
  \item Type names always end in 'type'.
  \item SET type names always end in 'SETtype'
  \item A distinction has been made between SET OF and \{EFF\}SET OF (only
as a comment and in the naming of attributes). \{EFF\}SET OF is used  for set
attributes that are only sets because it makes analysis more efficient.
Example: synvpefs in the clauserecord. 

This distinction is somewhat ad hoc, and not general enough. Lisette pointed
out to me that all sets that are not inherent are EFFSET-s. Furthermore, it
would be nice if inherent attributes and noninherent ones are formally
distinguished, and if changing the value of an inherent attribute in a rule
would be forbidden by the rule compiler and/or executer. Furthermore, if
a set is used as an inherent value, then this is used to indicate a set
of potential values. In these cases always an actual value should be selected
as well. It would be nice if one could guarantee this somehow. The notion
of EFFSET can disappear then, if a formal distinction between inherent and
noninherent attributes is made.
  \item EFFSET type names always end in EFFSETtype.
  \item Attributes of type EFFSET always end in 'efs'.  
  \item These conventions do not necessarily hold for parts taken over from
morphology. Parts taken over from morphology are marked with \{* morph *\},
parts that have been taken over from morphology, but have been changed
are marked with \{* morph ++*\}
\end{itemize}


Basically, the syntax of doc. 128 has been used, although there are 
some deviations from the syntax as described there, but in accordance
with the actually implemented syntax.
This document will have to be adapted to the definitive syntax for
DOMAIN T, as soon as this is known.
To allow for categories without a record, the record NORECORD has been 
defined. This is temporary and should be changed.

It has been decided in a discussion on the domain of morphology 
to start all basic categories by the prefix {\em B }. This has not been done
for categories that were present at that time already, so that there is
no consistency on this point in this document. A consequence of this 
decision would be that e.g. PREP should be called BPREP etc.
An alternative to this convention would be to prefix only those basic
categories by {\em B } for which there is also a nonbasic category. One author
of this document (Jan O) proposes to drop the decision made, and to adopt
the alternative convention.

There is a difference between the authors of this document on the status of
attributes.  Franciska de Jong adds attributes, even when this is not strictly
necessary to distinguish certain forms systematically, to represent properties
of categories (example: the attribute {\em definite} in BPERSPROrecord will
always have the value {\em true }). Jan Odijk has assumed attributes only
when this is required to make a subclassification of a category.
A decision should be made on this issue.

For conveniency, we have decided to document certain attributes and values that
are taken  over from morphology here as well, at least partially, and only to
the extent that they have consequences for MRULES.

\section{Polarity}

In this section the (DOMAIN T part of the ) apparatus to account for polarity
phenomena is described. It is based on a theory of polarity that is ad-hoc,
and for which no study has been made. For this reason the actual attributes,
values etc. have not yet been incorporated into Domain T, but this, or an 
alternative, must certainly be done.

Polarity phenomena are the phenomena where certain words, phrases
require the presence of other elements, or constructions, in particular
w.r.t. negation, nonnegation, yes-no questioning. A typical
example is, e.g. English {\em any }, that can occur only in negative, yes-no
and a limited number of other contexts. A case of an element  requiring a 
positive polarity environment is e.g. Dutch {\em nauwelijks } (cf. {\em Er is
nauwelijks (*n)iemand gekomen }). 

The distance between the element that imposes requirements upon polarity
and the element that satisfies this restriction, is typically unbounded.
For this reason, (given the limited power of M-rules)
I assume a mechanism of upward copying of attribute values  
throughout the structure.

I assume that there are two attributes:
\begin{itemize}
  \item polarity\_requirement: polaritytype:omegapolarity
        It encodes the polarity requirements imposed by certain 
        categories 
  \item polarity\_environment: polaritytype:omegapolarity
        It encodes the type of environment available w.r.t polarity
\end{itemize}

where the type {\em  polaritytype} is defined as follows:

\begin{description}
  \item[polaritytype] =    (positive, negative, omegapolarity);
\end{description}

The following RECORDS have the first attribute:
\begin{itemize}
  \item 
BADJrecord, SUBADJrecord, ADJrecord, ADJPrecord, ADJPPROPrecord, BADVrecord,
SUBADVrecord, ADVrecord, ADVPrecord, ADVPPROPrecord, ADVSENTrecord, ?ASPrecord,
CLAUSErecord, ?CONJrecord, BDETrecord, DETPrecord, BINDEFPROrecord,
BNOUNrecord, SUBNOUNrecord, NOUNrecord, CNrecord, NPrecord, NPPROPrecord, PREPPrecord, BQrecord, QPrecord, SENTENCErecord, ?THANPrec
ord,
BVERBrecord, SUBVERBrecord, VERBrecord, VERBPPROPrecord
\end{itemize}

The following records have the second attribute:
\begin{itemize}
  \item
CLAUSErecord, SENTENCErecord
\end{itemize}

As soon as an element is introduced, then the value of its attribute
{\em polarity\_requirement} is copied upward to the highest node in the
S-tree available at that point, but never higher than SENTENCErecord
or CLAUSErecord.
If the element introduced has a CLAUSErecord or SENTENCErecord itself, then
the value of this attribute is copied upward only if the 
attribute {\em polarity\_environment } has the value {\em omegapolarity }.

If the value copied upward into a SENTENCE- or CLAUSErecord is {\em  negative},
and the value of polarity\_environment or of polarity\_requirement of the
SENTENCE or CLAUSErecord is {\em positive }, then the rule that introduces this
element is not applicable. Similarly, if the value copied upward is {\em
positive }, and the value of polarity\_environment or of polarity\_requirement
of the SENTENCE or CLAUSErecord is {\em negative }, then again the rule is not
applicable. 

No SENTENCE can get dominated by UTT, the topcategory of Rosetta3,
unless:
\begin{itemize} 
  \item SENTENCErecord.polarity\_requirement = omegapolarity OR
  \item (SENTENCErecord.polarity\_requirement = positive AND\\
         SENTENCErecord.polarity\_environment =/=  negative )  OR
  \item (SENTENCErecord.polarity\_requirement = negative AND\\
         SENTENCErecord.polarity\_environment  = negative )  
\end{itemize}

In English (and perhaps in Spanish as well?) the introduction of a
an element that has the attribute {\em polarity\_requirement} set to 
{\em negative} as a subject
or as a shifted element resets the attribute polarity\_environment, if it
is specified as {\em negative }, to {\em omegapolarity }. Cf.
{\em *Anyone/You didn't see anyone }.

The value for the attribute {\em polarity\_environment} is set by several
rules, in particular:
\begin{itemize}
  \item the rule that introduces NEG (polarity\_environment:= negative)
  \item the mood determination rule that sets mood to {\em yes-nointerrogative }
        (polarity\_environment:= negative)
\end{itemize}

There are still some problematic aspects here, e.g. the verb {\em hoeven } in
Dutch can only be saved by a negation, not by a yesnointerrogative. (cf.
{\em *Hoeft hij te komen}. However, the negation must also be in the same
sentence (cf. {\em *hij zei niet dat hij hoefde te komen }, so maybe this is
a totally different phenomenon. It looks more like restrictions as imposed
on words like {\em nadie, nada } in Spanish, that require the presence
of a negative element very locally (in the same sentence?).
Concerning English {\em any }, it can also appear (in a different meaning,
universally quantifying) in the context of modal verbs. It is unclear how this
should be accounted for. 

An example derivation( where polarity\_environment is abbreviated to {\em env }
and polarity\_requirement to {\em req } for the sentence 
{\em John did not say that he saw anyone}:
\begin{description}
  \item[Subordinate clause]

\begin{itemize}
\item CLAUSE[x1 saw x2] + anyone (req:negative) $\rightarrow$
\item SENTENCE\{req:neg;env:omega\}[x1 saw anyone] + he  $\rightarrow$
\item SENTENCE\{req:neg;env:omega\}[he saw anyone] $\rightarrow$
\end{itemize}
 
   \item[Main clause]
\begin{itemize}
  \item CLAUSE[ x1 did say x2] + subordinate clause $\rightarrow$
  \item CLAUSE\{req:neg;env:omega\}[x1 did say SENTENCE\{req:neg;env:omega\}
          
          [he saw anyone]  ]   + NEG    $\rightarrow$
  \item CLAUSE\{req:neg;env:{\em neg}\}[x1 did not say
SENTENCE\{req:neg;env:omega\} 
          
          [he saw anyone]  ]   + John $\rightarrow$
  \item John did not say that he saw anyone
\end{itemize}
\end{description}

\subsection{General Remarks}

As decided in the morphology group, the differences between BX, SUBX and
X categories are:
\begin{itemize}
  \item BX categories have a KEY attribute. SUBX and X categories do not.
  \item X categories have noninherent attributes , BX and SUBX categories
        do not.
\end{itemize}

The X Categories for which there is no SUBX, but only a BX and an X (examples:
BPERSPRO/PERSPRO, BPROPERNOUN/PROPERNOUN) do not have a KEY attribute either.

In particular:
\begin{description}
  \item[mood, soort] These are determined in R2 in the startclauserules. This
information, however, is not necessary until a very late stage. In R2
no real use of this information is made until the time rules. There they
are only refined(made more specific),  with the exception of the values
impenk and impmv which are used to block the formation of imperative
subordinate sentences. But we might just as well block formation of such
sentences later in the derivation (in the moodddetermination rules).
The attribute soort is used furthermore in Orderrules, advsubstrules, 
inversion rules and in endclauserules. The R3 equivalents of these rules
either need not refer to soort, or apply at a point where the equivalent of
{\em soort} is known. 

The attribute mood is used in the voice rules (to prevent passivisation
of imperatives). This restriction can be imposed later as well (e.g. in
the mooddetermination rules, or perhaps in the clauseformation rules).
Furthermore it is used in the tijdregels (e.g. to prevent subordinate 
imperatives) and in the shift and endclause rules. None of these cases
require that mood be known already in VERBPPROPrec. 

Maybe the attributes are present in R2 out of efficiency considerations.
If so, then it might be necessary, useful to introduce them here as well.
  \item[tijd] It is present in R2 from the startclrules, but it is referred to 
(in fact assigned a value) only in the time rules. So it is not necessary
before the clauseformation rules in R3. (or for efficiency??)
\end{description}

\section{CATEGORIES}

The categories starting with the prefix {\em OPEN } or {\em CLOSED } are the
categories that are the export of the XPPROPtoXPPROP subgrammars. In
accordance with doc.  101 on Control Expressions these subgrammars are
adapted in such a way that their exportcategories do not intersect with
their input categories.'OPEN' categories have their attribute PROsubject
set to true, CLOSED categories have this attribute set to false.
Notice that the prefixes 'OPEN' and 'CLOSED' have a meaning differing from
what they usually (and in R2) have. 'CLOSED' categories can still contain
VAR-s, and 'OPEN' categories need not contain VAR-s.
'OPEN' categories either have BIGPRO as their subject, or have a NPVAR as
their subject with detnpmood = prosubjxpmood; 'CLOSED' categories either have
an NP as subject, or an NPVAR with detnpmood =/= prosubjxpmood.
