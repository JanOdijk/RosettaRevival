\section{Introduction}
During the last weeks several scenario's for extending the Rosetta computer 
facilities emerged from discussions in which Carel Fellinger, Rene Leermakers, 
Jan Stevens and myself participated. In the next sections I will describe 
these scenario's in detail,
that is, the actions that have to be done, the ``time-path'' that has to be 
followed and the man-power that is required. In each of the scenario's 
sketched in sections 2 and 3 an extension is proposed in the form of a network 
to which a number of workstations are connected. This seems to be the best 
answer to the problems caused by the lack of CPU capacity in the Rosetta 
project. But the development 
of the Rosetta system and the system itself are not only very CPU-intensive
they also need powerful windowing facilities ( multiple windows, 
large windows, etc). This requirement too is met by the workstation solution
in which each station is equipped with large screens.  
\section{Scenario 1: VMS-LAVC}
In scenario 1 the computer power is
enlarged by building a L(ocal) A(rea) V(ax) C(luster). In this cluster the
VAX 11/780 ( the Rosetta VAX ) and the MicroVax can be used. The 780 is used 
as the central node (`boot-member') of the cluster. 
Furthermore a number of VS2000 workstations ( say 15 )
are added. Each of these workstations has the CPU power of a VAX 11/780 
and can have 8Mb local main memory.

In this scenario no investment in man-power is required for the transition to 
the new configuration because all software that has been developed can be 
used without further changes. 

At the moment, however, LAVC is a rigid network, e.g. only one VAX 11/780 and
15 workstations can be connected to the network.

\section{Scenario 2: UNIX-SUN}
In the second scenario the VAX 11/780 is replaced by a network consisting of
2 SUN file servers and 15 SUN work stations. A SUN work station is a
1.5 (SUN 3/50), a 3 Mips (SUN 3/60) or a 10 Mips (SUN 4) machine with 
4Mb, 4Mb or 8 (-32) Mb respectively.

A transition from VAX to SUN also implies a transition from the VMS 
operating system to the UNIX operating system. The software that has
been developed under VMS cannot simply be used under UNIX. 

For a number of good reasons the Rosetta
system has been written in a version of PASCAL which deviates from the ISO-
PASCAL definition in some (well defined) points ( cf. Rosetta document 64,
`Programming Standards'). The different kinds of
deviations that should be distinguished here are: 

\begin{enumerate}
  \item The software layer in the Rosetta system ( Kernel ) that makes use of 
        {\em system 
        calls} and VMS packages, e.g. window manipulation, indexed file I/O (RMS), 
        string manipulation, memory usage etc..
  \item The extensions of VMS PASCAL that are used to implement a Modula2-like
        {\em module} mechanism. E.g. usage of the INHERIT, ENVIRONMENT mechanism and the
        underscore character in identifiers
  \item The (very limited) usage of the Typecast operator in order to implement 
        a kind of ADA-like {\em generic} mechanism.
  \item Those pieces of the software in which software developpers
        have departed ( by accident or on purpose ) from the Rosetta PASCAL 
        standard. E.g. the OTHERWISE clause, complex function return types and  
        the usage of STATIC variables ( in compilers ). 
\end{enumerate}

We can imagine two possible scenario's for a transition to the SUN system
\begin{description}
  \item[Scenario 2a] A conversion from VMS PASCAL to SUN PASCAL.
  \item[Scenario 2b] A conversion from VMS PASCAL to Modula-2.
\end{description}

The advantage of Scenario 2b with respect to Scenario 2a is that we will 
ultimately end up with a more portable system. This can be demonstrated by 
looking at points 1 to 4 of the deviations from ISO-PASCAL. If we would have
chosen Modula-2 as the Rosetta programming language two years ago a conversion
from VMS to SUN would only require a reprogramming of the point 1 software.
No effort would have been required to transport the software mentioned in points 2 
to 4 because they can be implemented using regular Modula-2 language 
constructions ( it is good to mention that the most important reason for not 
chosing Modula-2 at that time was the fact that we could not lay hands on 
good Modula-2 compilers under VMS). 
Another advantage of Modula-2  is the fact
that the Modula-2 compilers on the SUN allow characters to have an 8-bit ASCII 
value. In standard UNIX ( Berkeley ) PASCAL
the character range is limited to 7-bit ASCII values. At first sight this may 
seem a minor difference, from the Rosetta point of view, however, this 
difference is crucial. In the Rosetta system we need a multinational character 
set and all eight bits of a byte ( 256 possible character values ) are needed 
to implement such a set. 

We estimate that the amount of man-months that will be necessary for 
both scenario's is as follows:

\begin{verbatim}
                              Scenario 2a               Scenario 2b

Deviation 1  (= Kernel)          7                          7
Deviation 1*                     8                          8
Deviation 2,3,4                  3                          6
Lingware                         -                          1
                                ___                        ---
Total                           10                         14
Total*                          11                         15
\end{verbatim}

( In the items marked with a `*' symbol a Memory Management Module will be 
  developed which is better adapted to the needs of the Rosetta application )


The Rosetta Kernel consists of a number of modules with a well defined 
interface. Because the kernel makes use of system calls for process creation,
communication, I/O, etc. it is necessary to have a lot of knowledge of the
insides of the UNIX operating system. 
As far as the Rosetta project is concerned, this knowledge is not sufficiently 
present (yet). For a proper use of the system calls also the interface 
between the
programming language ( PASCAL or Modula-2 ) and the operating system
must be investigated. Furthermore, it will be necessary to use the software 
package C-ISAM, written in the C programming language, as a substitute
for RMS. Therefore the interface between
our programming language and C must be studied. If these steps have been
taken reprogramming the Rosetta kernel with a fixed interface seems to be a 
straight-forward task.

The remaining Rosetta software that must be converted can be divided into 2 
parts, one part is the regular PASCAL source code, the other part is the
code generation software of our compilers. For the first part it is imaginable
that a special kind of tool can be constructed which can do the task 
automatically, the second part, however, must be done by hand. We expect no
serious problems with the conversion of this part of the Rosetta software.

In Scenario 2b also parts of the Lingware have to be changed because parts
of some kind of rules are plain PASCAL. These pieces PASCAL code are not 
checked by our
self-made compilers, they just copy the code into the PASCAL modules which 
are generated. If we decide to use Modula-2 these PASCAL parts have to be converted 
into Modula-2.

Besides a port of the Rosetta software also a software development environment 
has to be created on the SUN. The most important part of that task ist to
transport the Rosetta software management system RBS from VMS to UNIX. 
This comes down to rewriting the RBS system almost completely because it has 
been written in DCL, the command language of VMS.
We think that it is 
possible for an {\em RBS specialist} to accomplish this task within 6 months.
But there is even more to it for the RBS system is adaptable to a specific 
application. Therefore the
specific adaptations that have been made for the Rosetta application 
( e.g. the command files in which compilers are invoked, the
editor command files, etc.) must also be rewritten. 
It should be possible to make the Rosetta adaptations of RBS on the SUN 
within 1 month.

Finally, it must be remarked that the estimates of the time needed for a
straight-forward conversion of the Rosetta software (deviations 2,3,4) will
be fairly accurate. The estimate for reprogramming the Rosetta Kernel, however,
is based on the expectation that we will meet no serious or insurmountable 
difficulties and that the kernel can be reprogrammed without changing its 
interface. Furthermore, the estimate does not include a possible learning 
period, that is, the time needed to study the Rosetta kernel as it is and the 
time needed to become conversant with the UNIX system calls.
\section {More detailed specification}
\subsection{Deviation 1: Kernel}
\begin{itemize}
  \item {\bf STR:} String manipulation routines : 1 week.
  \item {\bf FILES:} Text file manipulation : 2 weeks.
  \item {\bf WINDOWS:} Window manipulation. On the Sun the large X-WINDOWS package 
                 must be used : 2 months.
  \item {\bf MB:} Process communication by means of mailboxes : 2 weeks.
  \item {\bf PC:} Process control : 2 weeks.
  \item {\bf MEMORY:} Memory management : 1 week - 1 month.
  \item {\bf RMS:} Index sequential File I/O. : 2 month
  \item {\bf Miscellaneous:} CLOCK, ABORT..: 1 month.
\end{itemize}

\subsection{Deviation 2,3,4: $\rightarrow$ PASCAL}
\begin{itemize}
  \item Conversion of the standard Rosetta software : 1 month.
  \item Adaptation of the code generation part of the compilers : 2 months.
\end{itemize}

\subsection{Deviation 2,3,4: $\rightarrow$ Modula-2}
\begin{itemize}
  \item Conversion of the standard Rosetta software : 3 months.
  \item Adaptation of the code generation part of the compilers : 3 months.
\end{itemize}

\section{A possible time path for scenario's 2a and 2b}
In this planning the conversion to SUN UNIX is performed by members of a 
Software House. The idea is to have 5 activities some of which can be
performed in parallel:
\begin{enumerate}
  \item Creation of a complete Rosetta system under VMS ( by the Rosetta
        project members ).
  \item Conversion activity between VMS-PASCAL and SUN-Modula-2. This will
        be done by a team consisting of three Software House employees and 
        a member of Rosetta project who will join the team in a later stage for 
        the code generation parts of the compilers.
  \item RBS development for SUN ( one Rosetta project member in a later
        stage assisted by one Software House employee ).
  \item Development of the M-grammars on the VMS VAX.
  \item Conversion of the Lingware.
\end{enumerate}
